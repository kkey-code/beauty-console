# 订单列表压测与 N+1 查询优化对比报告

## 一、测试目标

验证 `/admin/service-orders?page=1&pageSize=20` 在优化前后的性能差异，并确认优化没有引入接口错误或响应结构变化。

本次只优化读接口：把分页结果中逐订单查询客户、订单明细、明细员工和支付记录的 N+1 查询，改为按当前页 ID 集合批量查询后在内存中分组装配。

## 二、测试环境

- 测试日期：2026-07-15
- 操作系统：Windows + WSL 2 + Docker Desktop
- 应用：Spring Boot 3.5.16、Java 17
- 数据库：MySQL 8.0 Docker 容器
- 缓存：Redis 7.4 Docker 容器
- 压测工具：官方 `grafana/k6:latest` Docker 镜像
- 测试数据：100 条订单、100 条订单明细、100 条支付记录；逻辑删除后列表可见订单 98 条
- 分页大小：20 条
- 压力模型：固定到达率 20 请求/秒，持续 30 秒
- 验证条件：HTTP 失败率小于 1%、P95 小于 1000ms、业务检查成功率大于 99%

## 三、复现方式

压测脚本位于 `performance/order-list.js`。脚本会在 `setup` 阶段登录一次，并在每次请求中检查 HTTP 状态、统一响应业务码和分页数据结构。

```powershell
# 启动本地环境
docker compose up -d --build

# 第一次压测需把项目种子数据导入本地测试库
docker compose cp beauty-console/store-server/src/main/resources/sql/seed_100_rows.sql mysql:/tmp/seed_100_rows.sql
docker compose exec -T mysql sh -c "mysql -ustore_app -pstore_app_change_me db_platform < /tmp/seed_100_rows.sql"

# 默认每秒 20 请求、持续 30 秒
docker run --rm -v "${PWD}\performance:/scripts:ro" grafana/k6:latest run /scripts/order-list.js
```

也可以覆盖参数：

```powershell
docker run --rm `
  -e RATE=50 `
  -e DURATION=60s `
  -e BASE_URL=http://host.docker.internal:8080 `
  -v "${PWD}\performance:/scripts:ro" `
  grafana/k6:latest run /scripts/order-list.js
```

## 四、优化前问题

原来的 `pageOrders` 先查询订单分页，再对当前页每一条订单调用 `toVO`：

1. 分页总数和分页记录：2 次 SQL。
2. 每个订单查询一次客户：N 次 SQL。
3. 每个订单查询一次订单明细：N 次 SQL。
4. 每个订单批量查询一次该订单明细涉及的员工：N 次 SQL。
5. 每个订单查询一次支付记录：N 次 SQL。

因此总查询数约为：

```text
2 + 4N
```

当 `pageSize=20` 且每条种子订单都有明细和员工时，约为 82 次 SQL。页大小越大，SQL 次数线性增长，数据库连接和网络往返会成为主要开销。

## 五、优化方案

优化后的流程：

1. 正常执行分页总数和分页记录查询。
2. 收集当前页所有订单 ID 和客户 ID。
3. 一次批量查询全部客户。
4. 一次批量查询全部订单明细。
5. 从明细中收集员工 ID，一次批量查询全部员工。
6. 一次批量查询全部支付记录。
7. 使用 Map 按客户 ID、订单 ID、员工 ID 分组，在内存中装配 `ServiceOrderVO`。

总查询数固定约为 6 次，不再随当前页订单数量线性增加。分页 20 条时，查询次数由约 82 次降为约 6 次，下降约 92.7%。

## 六、实测结果

两轮均使用同一 Docker 环境、同一批数据、同一脚本和同一压力参数。

| 指标 | 优化前 | 优化后 | 变化 |
| --- | ---: | ---: | ---: |
| 完成的列表迭代 | 600 | 601 | 固定到达率均完成 |
| 检查成功率 | 100% | 100% | 持平 |
| HTTP 失败率 | 0% | 0% | 持平 |
| 平均响应时间 | 40.63ms | 16.20ms | 下降约 60.1% |
| 中位数 | 30.87ms | 13.14ms | 下降约 57.4% |
| P90 | 62.42ms | 20.28ms | 下降约 67.5% |

## 八、面试时怎么讲

**问：什么是 N+1 查询？**

答：先用 1 次查询拿到 N 条主记录，再为每条主记录分别查询关联数据，最终产生 1+N 甚至 1+多个 N 次查询。本项目订单分页原来对每个订单分别查客户、明细、员工和支付，20 条订单大约需要 82 次 SQL。

**问：怎么优化的？**

答：先收集当前页关联 ID，用 `IN` 批量查询客户、订单明细、员工和支付，再按 ID 分组装配 VO。查询次数从随页大小增长变成固定约 6 次。

**问：为什么不直接把所有表写成一条 JOIN？**

答：订单、明细、支付都是一对多关系，直接同时 JOIN 容易产生明细与支付的笛卡尔放大，分页和总数也更难保证正确。分批查询保持 SQL 简单，查询次数固定，装配逻辑也更可测试。

**问：压测结果能代表生产性能吗？**

答：不能直接等同于生产。本次是本机同条件 A/B 对比，能证明代码改动降低了查询往返和本地延迟；生产还需要使用接近真实的数据规模、机器配置、网络、并发模型和监控指标重新压测。

## 九、控制台与项目耗材补充优化

### 9.1 问题与方案

| 场景 | 优化前 | 优化后 |
| --- | --- | --- |
| 控制台 | 进入页面并发 7 个 HTTP 请求 | 1 个聚合接口，Redis 按用户/角色缓存 30 秒 |
| 项目耗材 10 条分页 | 约 21 次 SQL | 约 3 次 SQL：分页、批量项目、批量耗材 |
| 页面来回切换 | 旧请求继续执行，旧响应可能覆盖新状态 | 新请求/离开路由/销毁组件时取消旧请求，并用序号丢弃过期响应 |
| 网络故障 | Axios 最长等待 10 分钟 | 30 秒超时，加载期间禁止重复触发 |
| 前端静态资源 | 文本资源未压缩、每次重新验证 | gzip、带哈希静态文件缓存 30 天、代理 keep-alive |

### 9.2 本地 Docker 实测

| 指标 | 结果 |
| --- | ---: |
| 控制台强制刷新 | 128.2ms |
| 控制台缓存读取 10 次 | 平均 18.0ms / Pseudo-min 12.7ms / max 31.1ms |
| 项目耗材分页 10 次 | 平均 38.5ms / min 18.2ms / max 158.4ms |
| vendor JS 构建体积 | 2.22MiB |
| vendor JS gzip 体积 | 685.93KiB，约减少 69.8% |
| 页面切换验证 | 项目耗材 -> 库存耗材 -> 客户档案 -> 项目耗材，正常，0 个控制台错误 |

以上接口耗时是本机 Docker 单机抽样，不是完整 k6 A/B 压测，不能与订单列表的 20 请求/秒报告混为一组生产结论。公网首屏还受客户端与香港服务器之间的线路、带宽和浏览器缓存影响；本轮主要通过减少传输量和请求次数降低这一部分开销。

### 9.3 Redis 的边界

Redis 不是“项目接入后所有接口自动变快”。本轮只缓存满足以下条件的数据：读取频繁、计算需要多次查询、允许 30 秒内短暂不一致、能明确设计用户隔离与失效策略。项目耗材分页包含多种查询条件且要求较强实时性，因此选择批量 SQL 而不是缓存所有分页组合。
