# 问题清单

本文只单独提取项目过程中遇到的问题，不展开解决方法。详细原因、处理方式和验证结果见 `问题记录.md`。

## 一、项目结构与代码分层

1. 项目结构混杂，DTO、VO、Entity、Controller、Service、Mapper 边界不清。
2. Entity 直接接收前端请求，接口入参和数据库模型耦合。
3. Controller 写入过多业务逻辑，后续扩展和测试成本高。
4. 公共工具、异常、统一返回结果散落在业务模块。
5. Service/Impl 目录大小写不统一，在不同系统上容易出现路径问题。
6. 服务订单 Controller 业务偏重，创建、完成、取消等流程不够集中。
7. 预约状态流转散落在 Controller，容易绕过合法状态机。

## 二、数据库与实体字段

1. 代码字段和数据库字段不一致，例如 `debtStatus`、`staffId`。
2. 新增 Entity 字段时，数据库、DTO、VO、Mapper、接口文档容易漏同步。
3. SQL patch 绑定具体库名，不利于多环境执行。
4. 预约号、订单号、收款流水号早期按时间生成，高并发下有重复风险。
5. 业务编号缺少数据库唯一索引兜底。
6. 逻辑删除和自动填充时，部分表的 `update_time` 可能为空。

## 三、登录认证与权限

1. 登录链路中 Controller、Service、JWT 生成职责不清。
2. JWT Claim 名称和拦截器读取名称不一致。
3. 配置中的 token 请求头名称和前端传参名称不一致。
4. 密码字段名是 `password_hash`，但早期实际保存明文。
5. 登录日志存在输出密码的风险。
6. 拦截器手动 `new` 导致 Spring 注入失效。
7. `addInterceptors` 方法误加 `@Bean` 会导致启动失败。
8. 角色权限过宽，不同角色可能看到或调用不该访问的接口。
9. JWT 里携带权限后，权限变更不能实时失效。
10. 账号权限早期不能自由分配权限点。
11. “恢复默认权限”按钮语义像清空权限，用户难以区分。

## 四、接口入参与分页

1. 新增、修改接口缺少 `@RequestBody`，JSON 请求体无法绑定 DTO。
2. 路径 ID 和请求体字段边界不清，容易被前端覆盖。
3. 分页查询空条件直接拼入 SQL，可能导致查询结果异常。
4. 所有列表分页早期没有真正启用 MyBatis-Plus 物理分页。
5. 分页排序不稳定，翻页时数据顺序可能漂移。
6. 接口文档不完整，字段、状态码、DTO、VO、业务规则缺失。

## 五、对象复制与数据返回

1. `BeanUtils.copyProperties` 浅拷贝可能覆盖不该覆盖的字段。
2. DTO、Entity、VO 字段不完全一致时，复制结果容易丢字段或误写字段。
3. 订单明细查询存在 N+1 查询风险。
4. 工作台待服务订单、库存预警、近期预约展示数量和排序规则不明确。

## 六、订单、收款与金额

1. 订单金额早期缺少强校验，明细金额和订单总额可能不一致。
2. 收款记录早期只是保存流水，没有同步反写订单已付、欠款和支付状态。
3. 新增收款、作废收款并发执行时，可能基于旧金额计算导致覆盖。
4. 已取消订单仍可能被收款或完成。
5. 已退款订单存在重复退款风险。
6. 订单取消、完成、删除等状态规则不够明确。

## 七、库存与耗材

1. 库存流水早期没有稳定同步库存数量。
2. 出库数量可能超过当前库存。
3. 库存扣减并发时存在错账、覆盖或超扣风险。
4. 库存扣减重试路径覆盖不足。
5. 订单完成后自动扣减哪些耗材的规则不明确。
6. 服务项目缺少耗材配置时，订单完成链路不清晰。
7. 已完成订单取消时早期缺少库存回滚规则。
8. 本地库存耗材数据偏模板化，不方便真实联调。
9. 项目耗材关系为空，前端项目耗材页面显示暂无数据。

## 八、缓存、导出与报表

1. Redis 配置存在，但早期业务没有实际使用缓存。
2. Redis 不可用时缺少降级策略和运维监控。
3. EasyExcel 导出可能一次性加载过多数据。
4. 库存物品只支持导出，缺少导入能力。
5. 统计报表不足，例如库存消耗、低库存预警、员工业绩。

## 九、集成测试与真实联调

1. 早期测试以 Controller 和 Service 单元测试为主，缺少真实 Spring + MockMvc + 数据库链路。
2. 项目耗材维护接口缺少完整 CRUD 集成测试。
3. 权限清空和恢复默认的接口语义缺少集成测试。
4. H2 集成测试不能完全覆盖真实 MySQL 方言和 Redis 连接。
5. 真实 MySQL + Redis 联调入口不足。
6. 前端浏览器真实 E2E 测试还未纳入固定流程。

## 十、前端与联调

1. 前端最初仍放在苍穹外卖学习源码目录，不利于和当前项目统一维护。
2. 前端旧外卖模板视图、API、组件、资源残留过多，容易误改误引用。
3. 服务订单列表里订单明细 `staffId` 为空时可能触发 NPE。
4. 员工列表查询了数据库不存在的 `level` 字段。
5. 预约新增早期没有生成 `appointmentNo`。
6. 用户状态更新和删除失败时，业务失败容易被当成系统异常。
7. 不同角色前端菜单未过滤时，角色登录后可能看到无权限菜单并触发接口报错。
8. 前端目录当前不是 Git 仓库，前端改动无法随后端提交统一记录。

## 十一、审计与删除保护

1. 关键写操作缺少统一审计，误操作后难以追踪。
2. 审计表、审计实体和审计写入链路早期不存在。
3. 客户、员工、服务项目、库存物品、预约和订单直接删除可能破坏历史数据。
4. 已被预约、订单、库存流水或耗材关系引用的数据缺少删除保护。
5. 操作人 ID 早期可被前端传参伪造，审计和业务归属不可靠。

## 十二、执行环境与命令

1. PowerShell 不支持 MySQL `<` 输入重定向。
2. PowerShell 下 Maven `-D` 参数需要加引号。
3. 只跑 `store-server` 模块会丢失 `store-pojo` 等多模块依赖。
4. `-am` 搭配指定测试类会让上游模块报无匹配测试。
5. 当前 PowerShell 环境不支持 `&&` 作为命令连接符。
6. `rg` 搜索以 `-D` 开头的文本时需要使用 `--`。
7. Windows 下 Git 会提示 LF 将转换为 CRLF。
