# 问题记录

## 1. 文档说明

本文统一记录 `beauty-console` 项目从重构、业务开发、联调测试到本地执行过程中遇到的所有问题、原因分析、解决方法和后续注意事项。

后续新增问题统一追加到本文，不再拆分临时问题记录文件。

## 2. 项目结构问题

### 2.1 原问题

最开始项目结构比较混杂，DTO、VO、Entity、Controller、Service、Mapper 的边界不够清晰。后续开发时容易出现：

- Entity 直接接收前端请求。
- Controller 写太多业务逻辑。
- DTO 和 VO 字段不完整。
- 公共类散落在业务模块里。
- 后续新增功能不知道放在哪一层。

### 2.2 解决方法

重构为 Maven 多模块结构：

```text
store-common  -> 通用能力
store-pojo    -> DTO / VO / Entity / Enum
store-server  -> Controller / Service / Mapper / Config
```

这样拆分后：

- 请求参数统一进入 DTO。
- 返回前端的数据统一使用 VO。
- Entity 只负责数据库表映射。
- 公共工具、异常、返回结果放在 `store-common`。
- 业务接口和数据库访问集中在 `store-server`。

## 3. 数据库表字段和代码不一致

### 3.1 原问题

代码中存在字段，但数据库没有对应字段：

- `ServiceOrder.debtStatus`
- `ServiceOrderItem.staffId`

这会导致 MyBatis-Plus 查询或保存时报错。

### 3.2 解决方法

同步数据库结构，给对应表补字段：

```sql
alter table service_order add column debt_status tinyint;
alter table service_order_item add column staff_id bigint;
```

同时保留 SQL patch 文件，方便后续重新初始化数据库时执行。

### 3.3 经验

以后每次新增 Entity 字段，都要同时检查：

- 数据库表是否有字段。
- DTO 是否需要接收该字段。
- VO 是否需要返回该字段。
- Mapper XML 是否写了该字段。
- 接口文档是否同步更新。

## 4. 登录认证链路问题

### 4.1 原问题

登录链路中出现过多个问题：

- Controller 和 Service 职责混乱。
- JWT 生成位置不统一。
- JWT Claim 名称和拦截器读取名称不一致。
- 配置文件中的 token 请求头名称和代码读取名称不一致。
- 密码字段名是 `password_hash`，但实际保存的是明文。
- 登录日志可能输出密码。
- 拦截器手动 `new`，导致 Spring 注入失效。

### 4.2 解决方法

最终调整为：

```text
SysUserController
-> 接收登录 DTO
-> 调用 SysUserService.login()
-> Service 查询账号和员工信息
-> 校验密码
-> 生成 JWT
-> 返回 LoginUserVO
```

密码处理方式：

- 新密码统一使用 BCrypt 存储。
- 旧明文密码登录成功后自动升级为 BCrypt。
- 新增和修改用户时自动对非 BCrypt 密码加密。

JWT 处理方式：

- 登录时写入 `userId`、`staffId`、`username`、`roleId`、`roleCode`、`status` 等 claims。
- 拦截器从配置的请求头中获取 token。
- 解析成功后把 `staffId` 放入 `BaseContext`。
- 请求结束后清理 ThreadLocal。

### 4.3 经验

认证链路一定要保证三处一致：

```text
登录生成 token 的 claim 名称
-> 拦截器解析 token 的 claim 名称
-> 后续业务从 BaseContext 读取的用户身份
```

## 5. 拦截器注册问题

### 5.1 原问题

曾经在 `WebMvcConfiguration.addInterceptors` 方法上加了 `@Bean`，导致 Spring 启动失败：

```text
@Bean method 'addInterceptors' must not be declared as void
```

另外，使用 `new JwtTokenInterceptor()` 注册拦截器时，会绕过 Spring 容器管理，导致内部依赖注入失败。

### 5.2 解决方法

- 删除 `addInterceptors` 方法上的 `@Bean`。
- 拦截器使用构造方法接收 `JwtProperties`。
- 注册拦截器时确保依赖可用。
- 放行登录接口 `/admin/users/login`。
- 放行 `OPTIONS` 预检请求。

### 5.3 当前注意事项

当前配置中的请求头名称是：

```yaml
sky:
  jwt:
    admin-token-name: token
```

因此前端要传：

```http
token: Bearer xxx
```

如果前端希望传：

```http
Authorization: Bearer xxx
```

则需要把 `admin-token-name` 改成 `Authorization`。

## 6. Controller 入参绑定问题

### 6.1 原问题

多个新增、修改接口缺少 `@RequestBody`，导致前端用 JSON 提交时 DTO 无法正确绑定。

例如错误写法：

```java
public Result<Long> save(@Valid PaymentRecordDTO dto)
```

JSON 请求体不会自动绑定到 DTO。

### 6.2 解决方法

统一改成：

```java
public Result<Long> save(@Valid @RequestBody PaymentRecordDTO dto)
```

路径 ID 和请求体分开：

```java
public Result<Boolean> update(
        @PathVariable Long id,
        @Valid @RequestBody XxxDTO dto)
```

### 6.3 经验

- `GET` 查询：用普通 DTO 接收 query 参数。
- `POST` 新增：用 `@RequestBody DTO`。
- `PUT` 修改：用 `@PathVariable id` + `@RequestBody DTO`。
- `PATCH` 状态修改：可以用 query 参数，也可以用专门的状态 DTO。

## 7. 分页查询空条件问题

### 7.1 原问题

分页查询里直接写：

```java
wrapper.like(Entity::getName, dto.getName())
       .eq(Entity::getStatus, dto.getStatus());
```

当 `name` 或 `status` 是空值时，可能拼出错误或无意义的 SQL 条件，导致查询不到数据。

### 7.2 解决方法

使用 MyBatis-Plus 的条件参数：

```java
wrapper.like(StringUtils.hasText(dto.getName()), Entity::getName, dto.getName())
       .eq(dto.getStatus() != null, Entity::getStatus, dto.getStatus());
```

这样只有条件有值时才会拼接 SQL。

### 7.3 经验

分页查询必须遵守：

```text
字符串条件 -> StringUtils.hasText()
数字条件   -> dto.getXxx() != null
时间范围   -> beginTime/endTime != null
```

## 8. BeanUtils 复制问题

### 8.1 原问题

项目中大量使用：

```java
BeanUtils.copyProperties(source, target);
```

这个工具只能复制同名同类型字段，不能处理业务语义差异。

典型问题是从预约生成订单时，不能简单把 `Appointment` 复制到 `ServiceOrder`，因为订单需要：

- 订单编号。
- 订单金额。
- 支付状态。
- 欠款状态。
- 订单状态。
- 订单明细。

这些都不是简单复制能完成的。

### 8.2 解决方法

对语义不同的业务对象，改为显式赋值：

```text
预约客户 -> 订单客户
预约员工 -> 订单服务员工
预约时间 -> 订单服务时间
预约明细 -> 订单明细
明细金额汇总 -> 订单金额
```

并在保存前做金额校验。

### 8.3 经验

`BeanUtils.copyProperties` 只适合字段语义完全一致的场景。只要涉及金额、状态、编号、明细、库存、支付，都应该显式处理。

## 9. 订单金额校验问题

### 9.1 原问题

订单主表金额和订单明细金额可能不一致。例如：

- 明细合计是 300，主表原价写成 200。
- 优惠金额大于原价。
- 已付金额大于应收金额。
- 欠款金额和已付金额不匹配。
- 支付状态和实际金额不匹配。

如果不校验，后续支付、统计、欠款都会错。

### 9.2 解决方法

新增订单金额强校验逻辑：

```text
原价金额 >= 0
优惠金额 >= 0
优惠金额 <= 原价金额
应收金额 = 原价金额 - 优惠金额
已付金额 <= 应收金额
欠款金额 = 应收金额 - 已付金额
主表金额必须等于明细合计
支付状态必须和金额匹配
欠款状态必须和欠款金额匹配
```

### 9.3 经验

订单金额不要相信前端传值，后端必须重新计算和校验。

## 10. 支付记录业务问题

### 10.1 原问题

原来新增支付记录只是：

```java
BeanUtils.copyProperties(dto, record);
paymentRecordService.save(record);
```

这只能保存一条支付流水，不能代表真实收款业务。缺少：

- 订单是否存在校验。
- 订单是否已取消校验。
- 支付单号生成。
- 支付时间默认值。
- 操作人记录。
- 重复支付单号校验。
- 收款后反写订单已付金额。
- 退款后反写订单欠款。
- 支付状态刷新。
- 支付记录作废处理。

### 10.2 解决方法

新增支付记录时做完整业务处理：

```text
查询订单
-> 校验订单状态
-> 生成或校验 paymentNo
-> 保存 payment_record
-> 成功收款：增加 paidAmount，减少 debtAmount
-> 退款：减少 paidAmount，增加 debtAmount
-> 未确认记录：只保存，不影响订单金额
-> 作废记录：不能直接创建，只能通过作废接口处理
-> 刷新 payStatus 和 debtStatus
```

### 10.3 经验

支付记录不是普通 CRUD。支付记录必须和订单金额放在同一个事务里处理。

## 11. 库存流水和库存数量问题

### 11.1 原问题

库存流水最开始只保存 `inventory_stock_log`，没有同步修改 `inventory_sku.quantity`。

这会导致：

- 流水显示出库了，但库存数量不变。
- 低库存预警不准确。
- 后续库存统计不准确。

### 11.2 解决方法

新增库存流水时同步更新库存数量：

```text
查询库存物品
-> 读取 beforeQuantity
-> 根据 changeType 计算 afterQuantity
-> 校验 afterQuantity 不能小于 0
-> 更新 inventory_sku.quantity
-> 保存 inventory_stock_log
```

不同类型的计算规则：

| 类型 | 处理 |
|---|---|
| `stock_in` | 增加库存 |
| `return` | 增加库存 |
| `stock_out` | 减少库存 |
| `loss` | 减少库存 |
| `check` | 盘点后库存直接等于提交数量 |

## 12. 库存并发问题

### 12.1 原问题

当前库存扣减如果只是普通查询：

```java
InventorySku inventorySku = inventorySkuService.getById(id);
```

两个请求可能同时读到相同库存，分别扣减后造成库存错账。

例如库存 10，同时两个请求各出库 8：

```text
请求 A 读到 10
请求 B 读到 10
请求 A 更新为 2
请求 B 更新为 2
```

实际应该第二个请求库存不足。

### 12.2 解决方法

采用数据库行锁：

```java
InventorySku inventorySku = inventorySkuService.getOne(
        new LambdaQueryWrapper<InventorySku>()
                .eq(InventorySku::getId, dto.getInventoryId())
                .last("FOR UPDATE")
);
```

并确保外层方法有事务：

```java
@Transactional
public Result<Long> outbound(@Valid @RequestBody InventoryStockLogDTO dto) {
    ...
}
```

执行顺序：

```text
开启事务
-> 查询 inventory_sku FOR UPDATE
-> 校验库存
-> 更新库存
-> 写库存流水
-> 提交事务释放锁
```

### 12.3 当前状态

库存流水扣减已经落地数据库行锁方案。手工出库、订单自动扣库存都会通过同一套库存流水服务处理：

```text
查询 inventory_sku FOR UPDATE
-> 校验库存
-> 更新库存
-> 写库存流水
```

## 13. 订单自动扣库存问题

### 13.1 原问题

订单明细只有服务项目 ID，系统目前不知道每个服务项目需要消耗哪些库存物品。

现有结构：

```text
service_order_item
- service_project_id
- quantity
```

缺少服务项目和库存 SKU 的关系。

### 13.2 解决方法

已新增关系表：

```text
service_project_inventory
- id
- service_project_id
- inventory_id
- consume_quantity
- status
- remark
```

订单完成时：

```text
遍历订单明细
-> 根据 service_project_id 查询耗材配置
-> 计算实际扣减数量 = 订单项目数量 * consume_quantity
-> 对 inventory_sku 加 FOR UPDATE
-> 扣库存
-> 写库存流水
```

### 13.3 当前状态

订单完成接口已经接入自动扣库存。当前规则：

- 只有 `orderType = service` 的服务订单会自动扣耗材。
- 订单从待服务变成已完成时扣库存。
- 已完成订单再次点击完成不会重复扣库存。
- 库存不足时抛业务异常，订单状态不会更新为已完成。
- 没配置耗材关系的服务项目不能完成订单，避免订单完成但库存没有同步扣减。
- 耗材关系查询使用 Spring Cache 缓存，新增、修改、删除关系时清理缓存。

## 14. 枚举值不统一问题

### 14.1 原问题

部分数据库字段曾经保存中文值，代码中又按英文 code 处理，导致查询和展示不稳定。

涉及字段包括：

- `order_type`
- `payment_method`
- `inventory_stock_log.change_type`

### 14.2 解决方法

统一使用英文 code 保存，前端展示时由 VO 或枚举转换为中文名称。

示例：

| 字段 | 数据库存储 | 前端展示 |
|---|---|---|
| `payment_method` | `wechat` | 微信 |
| `payment_method` | `alipay` | 支付宝 |
| `change_type` | `stock_in` | 入库 |
| `change_type` | `stock_out` | 出库 |

### 14.3 经验

数据库建议保存稳定 code，不直接保存中文展示文案。

## 15. 角色字段调整问题

### 15.1 原问题

用户角色字段调整为 `role_id` 后，原代码中部分地方仍按旧的角色字符串或旧字段理解角色，可能导致 readonly 角色无法正确展示。

### 15.2 解决方法

统一按当前数据库字段 `role_id` 处理，并兼容六类角色：

```text
1 SUPER_ADMIN      超级管理员
2 STORE_MANAGER    店长
3 STAFF            普通员工
4 INVENTORY_ADMIN  库存管理员
5 FINANCE          财务/收银
6 READONLY         只读
```

同时保证登录返回 VO 中包含前端需要展示的 `roleId`、`roleCode`、`roleName`。


## 17. Redis 和 Spring Cache 问题

### 17.1 原问题

项目最开始引入了 Redis 配置和依赖，但业务代码没有实际使用缓存，只能算预留能力。

### 17.2 解决方法

当前已接入 Spring Cache：

```text
spring-boot-starter-cache
@EnableCaching
spring.cache.type=redis
```

服务项目耗材关系启用缓存：

```text
serviceProjectInventory:active
```

新增、修改、删除耗材关系时使用 `@CacheEvict(allEntries = true)` 清理缓存，避免订单扣库存读取旧配置。

### 17.3 注意事项

运行环境需要 Redis 可用。默认配置：

```yaml
spring:
  data:
    redis:
      host: localhost
      port: 6379
```

## 19. 提交前扫描发现的优化问题

### 19.1 角色权限过宽

原权限检查只按路径前缀判断，普通员工、库存管理员在部分接口上可能执行 `DELETE`。

解决方法：

- 权限检查改成角色、HTTP 方法、接口路径三者共同判断。
- 普通员工只允许日常业务的查询、新增、修改和状态流转，不允许删除客户、预约、订单。
- 库存管理员只允许库存相关查询、新增、修改和状态调整，不允许删除库存资料。
- 新增 `RolePermissionCheckerTest` 固化权限矩阵，防止后续改动放宽权限。

### 19.2 JWT 权限不能实时失效

原拦截器只读取 token 里的 `roleId/status/roleCode`。用户被禁用、角色被修改后，旧 token 在过期前仍可能继续使用旧权限。

解决方法：

- JWT 签名校验通过后，拦截器根据 `userId` 回查 `sys_user`。
- 校验数据库中的用户状态、角色、员工绑定是否仍和 token 一致。
- 状态或角色变化时返回未登录/登录信息变更，要求重新登录。
- 新增拦截器单测覆盖数据库禁用用户、角色变更、越权访问等场景。

### 19.3 库存导出可能一次性加载过多数据

原导出接口直接 `list` 查询所有匹配库存物品，再一次性写 Excel。数据量变大时容易占用过多内存。

解决方法：

- 导出前先 `count`，超过 10000 条要求缩小筛选条件。
- 使用分页查询分批写入 EasyExcel。
- 导出排序固定为库存 ID 升序，保证分页结果稳定。

### 19.4 服务订单扣库存规则不明确

原逻辑在服务项目没有配置耗材关系时会直接跳过扣库存，容易出现订单已完成但库存未减少的问题。

解决方法：

- 服务订单完成时，如果订单项目缺少服务项目或缺少启用的耗材关系，直接抛业务异常。
- 已完成订单再次完成仍保持幂等，不重复扣库存。
- 库存扣减仍统一走库存流水服务和 `FOR UPDATE` 行锁。

### 19.5 订单明细查询存在 N+1

原订单明细 VO 组装时，每条明细单独按 `staffId` 查询员工。

解决方法：

- 先收集订单明细中的 `staffId`。
- 使用 `listByIds` 批量查询员工，再映射回明细 VO。
- 新增单测验证不再逐条调用 `getById`。

### 19.6 操作人 ID 可被前端伪造

原库存流水 DTO 允许前端提交 `operatorId`。如果接口被绕过调用，可能写入错误操作人。

解决方法：

- 库存流水服务优先使用 `BaseContext` 中的当前员工 ID。
- 没有登录上下文时保留 DTO 操作人，方便单元测试和内部调用。
- 新增单测验证当前登录员工会覆盖请求里的 `operatorId`。

### 19.7 SQL patch 绑定具体库名

服务项目耗材关系脚本写死 `USE db_platform;`，与应用支持环境变量切换数据库的配置不一致。

解决方法：

- 移除脚本中的 `USE db_platform;`。
- 角色迁移脚本改为可重复执行，避免 `sys_role` 已存在但旧用户角色值未迁移时外键创建失败。

### 19.8 service/Impl 大小写路径问题

Windows 本地大小写不敏感，`service/Impl` 和 `service/impl` 容易混用；Linux/CI 环境会把它们当成不同路径。

解决方法：

- 使用 `git mv` 通过临时目录把服务实现包统一成小写 `service/impl`。
- 确保 Git 索引中只保留小写路径。

### 19.9 已完成订单取消缺少库存回滚

原规则只覆盖订单完成时自动扣减库存，没有覆盖已完成订单取消后的库存回滚。如果门店把已完成服务订单取消，库存会停留在已扣减状态，后续盘点和库存预警会不准确。

解决方法：

- 取消服务订单时先使用 `FOR UPDATE` 锁定订单，避免并发取消重复回滚。
- 只有已完成的服务订单需要库存回滚，待服务或待支付订单取消只更新订单状态。
- 回滚依据原始 `inventory_stock_log` 出库流水，而不是重新读取当前 `service_project_inventory` 配置。
- 同一库存 SKU 的多条原始出库流水会先汇总，再写入 `return` 类型库存流水。
- 如果已完成服务订单找不到原始出库流水，直接抛业务异常，避免静默漏回滚。

### 19.10 服务订单 Controller 业务偏重

服务订单 Controller 原来承担了分页、详情组装、新建、预约转订单、修改、取消和删除等业务逻辑。这样虽然能快速落地接口，但后续维护、单元测试和复用都会变困难。

解决方法：

- `ServiceOrderController` 保留接口入口和参数接收。
- 复杂业务下沉到 `ServiceOrderService` 和 `ServiceOrderServiceImpl`。
- Controller 单测改为验证接口委托关系，Service 单测覆盖完成扣库存和取消回滚等核心规则。

### 19.11 前端项目位置不统一

前端最初仍放在苍穹外卖学习源码目录下，不利于和当前后端项目一起维护，也容易让后续开发误改旧模板。

解决方法：

- 将当前改造后的前端复制到 `D:\_study\java\后台管理系统\beauty-console-web`。
- 前端目录与后端 `beauty-console` 并列管理。
- 复制时不携带 `node_modules`、`dist` 和临时日志，依赖在新目录重新安装。
- 前端运行默认访问 `http://localhost:8888/`，接口转发到 `http://localhost:8080`。

### 19.12 前端联调接口异常

前端联调时暴露了几类接口问题：服务订单列表里订单明细 `staffId` 为空会触发 NPE；员工列表查询了数据库不存在的 `level` 字段；客户和预约逻辑删除时 `update_time` 被填成空；预约新增没有生成 `appointmentNo`；用户状态更新和删除失败被当成系统异常。

解决方法：

- 订单明细组装员工姓名时先判断 `staffId`，没有员工时返回空员工名，不再访问不可接收空 key 的 `Map.of()`。
- `StaffMember.level` 标记为非数据库字段，避免 MyBatis-Plus 自动查询不存在的列。
- 增加 MyBatis-Plus 自动填充处理器，新增、更新和逻辑删除时统一填充 `createTime/updateTime`。
- 预约新增时服务端自动生成 `APTyyyyMMddHHmmssSSS` 编号；主服务员工优先使用前端传参，未传时使用当前登录员工，仍为空则返回业务提示。
- 用户状态更新同时兼容 body 和 query 参数，并把非法状态、用户不存在、删除失败转成业务异常。
- 用户删除前检查账号是否存在，并禁止删除当前登录账号。

### 19.13 分页、工作台和菜单权限问题

所有列表分页参数传到了后端，但没有注册 MyBatis-Plus 分页插件，导致 SQL 没有真正追加分页条件。工作台固定取前几条数据时也受这个问题影响；不同角色前端菜单未过滤，角色登录后容易看到无权限菜单并触发接口报错。

解决方法：

- 增加 `MybatisPlusInterceptor` 和 `PaginationInnerInterceptor(DbType.MYSQL)`，统一启用物理分页。
- 各分页列表增加稳定排序，避免翻页时数据顺序漂移。
- 工作台待服务订单、近期预约和库存预警都只取前 5 条；库存预警使用后端 `lowStockOnly=true` 过滤。
- 前端增加角色权限工具，菜单、工作台请求和资源页操作按钮都按当前角色过滤。
- 种子数据补齐 6 个测试账号：`admin`、`manager`、`staff`、`inventory`、`finance`、`readonly`，密码统一为 `123456`。
- 新增 `patch_test_role_accounts.sql`，不重置数据库时可单独补齐这些测试账号。
- 前端项目对应修改位于 `D:\_study\java\后台管理系统\beauty-console-web`。

### 22.3 只跑 `store-server` 会丢失多模块依赖

问题：执行真实联调测试时，如果只使用：

```powershell
mvn -pl store-server "-Dstore.real-it=true" "-Dtest=AdminApiRealIntegrationTest" test
```

可能出现 `Type com.wkr.storepojo.entity.ServiceProjectInventory not present`。原因是只选择 `store-server` 模块时，当前 reactor 没有同时构建 `store-pojo` 等依赖模块。

解决：加 `-am`，让 Maven 同时构建被依赖模块：

```powershell
mvn -pl store-server -am "-Dstore.real-it=true" "-Dtest=AdminApiRealIntegrationTest" "-Dsurefire.failIfNoSpecifiedTests=false" test
```

### 22.4 `-am` 搭配指定测试类会让上游模块报无匹配测试

问题：加 `-am` 后，`store-common`、`store-pojo` 也进入 reactor。它们没有 `AdminApiRealIntegrationTest`，Surefire 会报：

```text
No tests matching pattern "AdminApiRealIntegrationTest" were executed
```

解决：补充：

```text
-Dsurefire.failIfNoSpecifiedTests=false
```

最终可用命令：

```powershell
mvn -pl store-server -am "-Dstore.real-it=true" "-Dtest=AdminApiRealIntegrationTest" "-Dsurefire.failIfNoSpecifiedTests=false" test
```

### 22.5 PowerShell 不支持当前环境里的 `&&`

问题：提交时直接执行：

```powershell
git add . && git commit -m "feat: ..."
```

在当前 PowerShell 版本里报错：`The token '&&' is not a valid statement separator`。

解决：拆成两条命令执行：

```powershell
git add .
git commit -m "feat: ..."
```

### 22.6 `rg` 搜索以 `-D` 开头的文本需要 `--`

问题：使用 `rg` 搜索 `-Dstore.real-it` 时，如果不加 `--`，`rg` 会把 `-D...` 当作命令参数解析。

解决：

```powershell
rg -n -g "*.md" -- "-Dstore\.real-it"
```
